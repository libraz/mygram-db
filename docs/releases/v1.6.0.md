# MygramDB v1.6.0 Release Notes

**Release Date:** 2026-04-15
**Type:** Feature / Bug Fix / Refactor
**Previous Version:** v1.5.4

---

## Overview

Version 1.6.0 is a major feature release that adds **BM25 relevance scoring**, **synonym dictionary expansion**, **search result highlighting**, **fuzzy search**, **FACET aggregation**, **V2 dump format**, and **MariaDB binlog replication support**. The release also includes comprehensive correctness fixes and security hardening across the entire codebase.

**Highlights:**

- **BM25 relevance scoring** — `SORT _score` ranks results by term frequency and document length
- **Synonym dictionary** — Automatic OR-group expansion from TSV synonym files
- **HIGHLIGHT clause** — Snippet generation with configurable tags and fragment control
- **Fuzzy search** — Levenshtein edit distance matching with `FUZZY` clause
- **FACET command** — Filter column value aggregation with document counts
- **V2 dump format** — Section envelope with per-section CRC32 and forward compatibility
- **MariaDB support** — Binlog replication with MariaDB GTID (domain-server-sequence)
- **246 files changed**, 21,037 insertions, 5,608 deletions

## New Features

### 1. BM25 Relevance Scoring

Sort search results by relevance using the BM25 ranking function.

**TCP Syntax:**
```
SEARCH articles "machine learning" SORT _score DESC LIMIT 10
```

**HTTP API:**
```json
{
  "q": "machine learning",
  "sort": {"column": "_score", "order": "DESC"},
  "limit": 10
}
```

**Parameters:**
- `k1 = 1.2` — Term frequency saturation
- `b = 0.75` — Document length normalization

BM25 scoring computes IDF (inverse document frequency) and TF (term frequency in normalized text) at query time. Requires `verify_text` to be set to `"ascii"` or `"all"` for TF computation.

**Files:** `src/index/bm25_scorer.{h,cpp}`, `src/query/result_sorter.cpp`

### 2. Synonym Dictionary

Automatically expand search terms to include synonyms defined in a TSV file.

**Configuration:**
```yaml
synonyms:
  enable: true
  file: "/etc/mygramdb/synonyms.tsv"
```

**Synonym File Format (TSV):**
```
car	automobile	vehicle
fast	quick	rapid	speedy
```

- One synonym group per line, tab-separated
- Bidirectional: searching for any term in a group expands to all terms (OR semantics)
- Maximum 20 synonyms per group
- Lines starting with `#` are comments
- All terms are normalized using the same rules as index entries (NFKC, width conversion)

**Files:** `src/query/synonym_dictionary.{h,cpp}`

### 3. Search Result Highlighting

Return text snippets with search terms highlighted.

**TCP Syntax:**
```
SEARCH articles "machine learning" HIGHLIGHT
SEARCH articles "machine learning" HIGHLIGHT TAG <strong> </strong>
SEARCH articles "machine learning" HIGHLIGHT SNIPPET_LEN 150 MAX_FRAGMENTS 5
```

**HTTP API:**
```json
{
  "q": "machine learning",
  "highlight": {
    "open_tag": "<em>",
    "close_tag": "</em>",
    "snippet_length": 100,
    "max_fragments": 3
  }
}
```

**Options:**
| Option | Default | Range | Description |
|--------|---------|-------|-------------|
| TAG | `<em>` / `</em>` | — | Open and close tags for highlighting |
| SNIPPET_LEN | 100 | 1–10,000 | Max code points per snippet fragment |
| MAX_FRAGMENTS | 3 | 1–100 | Max fragments joined by ellipsis |

Requires `verify_text` to be set to `"ascii"` or `"all"` (needs stored normalized text).

**Files:** `src/query/highlighter.{h,cpp}`

### 4. Fuzzy Search

Match terms within a specified Levenshtein edit distance.

**TCP Syntax:**
```
SEARCH articles "machine" FUZZY
SEARCH articles "machine" FUZZY 2
```

**HTTP API:**
```json
{
  "q": "machine",
  "fuzzy": 1
}
```

**Parameters:**
- Distance `1` (default): Matches terms within 1 edit operation (insertion, deletion, substitution)
- Distance `2`: Matches terms within 2 edit operations

Pre-filters candidates by length difference for performance. Checks whitespace-delimited words only.

**Files:** `src/utils/edit_distance.{h,cpp}`, `src/query/query_parser_clauses.cpp`

### 5. FACET Command

Aggregate distinct values of a filter column with document counts.

**TCP Syntax:**
```
FACET articles status
FACET articles category "search text" FILTER status = 1 LIMIT 10
```

**HTTP API:**
```
POST /articles/facet
```

```json
{
  "column": "category",
  "q": "optional search text",
  "filters": {"status": 1},
  "limit": 10
}
```

Returns distinct values sorted by count (descending). Optionally restricted to documents matching a search query with full clause support (AND, NOT, FILTER).

**Files:** `src/server/handlers/facet_handler.{h,cpp}`, `src/storage/filter_index.cpp`

### 6. V2 Dump Format

New section-envelope dump format with per-section CRC32 integrity checks and forward compatibility.

**Structure:**
```
┌──────────────────────────────────────────────┐
│ Fixed Header: Magic "MGDB" + Version 2       │
├──────────────────────────────────────────────┤
│ V2 Header: Size, Flags, Timestamp, GTID      │
├──────────────────────────────────────────────┤
│ Section 1: Config (envelope + CRC32)          │
├──────────────────────────────────────────────┤
│ Section 2..N: TableData (envelope + CRC32)    │
└──────────────────────────────────────────────┘
```

**Key improvements:**
- Per-section CRC32 integrity verification
- Unknown section types safely skipped (forward compatibility)
- `DUMP LOAD` auto-detects V1 or V2 format
- `DUMP SAVE` writes V2 format by default
- `DUMP INFO` reports format version and section types

**Files:** `src/storage/dump_format_v2.{h,cpp}`

### 7. MariaDB Binlog Replication Support

Full binlog replication support for MariaDB using MariaDB-native GTID format.

**Supported versions:** MariaDB 10.6+, 11.x (tested with 10.11 and 11.4)

**MariaDB GTID format:** `domain_id-server_id-sequence_number` (e.g., `0-1-42`)

**Configuration:**
```yaml
mysql:
  host: "127.0.0.1"
  port: 3306
  user: "repl_user"
  password: "your_password"
  database: "mydb"
  use_gtid: true
  binlog_format: "ROW"
  binlog_row_image: "FULL"
```

No additional configuration is needed — MygramDB auto-detects MariaDB vs MySQL from the server version string and uses the appropriate GTID format and binlog stream protocol.

**Files:** `src/mysql/mariadb_gtid.{h,cpp}`, `src/mysql/mariadb_binlog_stream.{h,cpp}`, `src/mysql/mariadb_event_parser.{h,cpp}`, `src/mysql/server_flavor.h`

## Bug Fixes

### 1. DUMP LOAD Premature Loading Guard Release

**Severity:** Medium

`HandleDumpLoad` called `loading_guard.Release()` before checking the `ReadDump` result and before restarting replication or rebuilding BM25 statistics. The guard now stays active through the entire success path and is released only after completion. On failure, the RAII destructor clears the flag automatically.

### 2. BM25 Cache Hit Out-of-Bounds Read

**Severity:** High — Could cause undefined behavior

When a search query hits the query cache, the cache-hit path skipped `GenerateTermInfos()`, leaving `term_infos` empty. BM25 scoring then accessed an empty vector out-of-bounds. Fixed by regenerating `term_infos` before BM25 scoring when empty.

### 3. HIGHLIGHT Rejected When Text Storage Disabled

**Severity:** Low

Added explicit validation: HIGHLIGHT clause is rejected with a clear error when `verify_text` is set to `"off"`, since highlighting requires stored normalized text.

### 4. Text Storage Not Disabled When verify_text Is Off

**Severity:** Low

`DocumentStore::SetStoreTexts()` was not called when `verify_text` is `"off"`, wasting memory storing text that was never used.

### 5. Missing `<condition_variable>` Include

**Severity:** Low

`query_cache.h` relied on transitive inclusion. Added direct include.

### 6. Security, Thread Safety, and Correctness Hardening

Multiple rounds of hardening across the codebase (see commits for full details):

- Password/credential masking in SHOW VARIABLES and CONFIG SHOW
- SQL injection prevention in InitialLoader and ConnectionValidator
- Path traversal detection improvements
- Null byte injection rejection in config paths
- Atomic conversions for cache and document store fields
- Mutex protection for shared state in binlog reader
- TOCTOU fixes in SnapshotScheduler
- Expected<T, Error> propagation for GTID parsing and field decoding

## Refactoring

### Search Pipeline Consolidation

Extracted reusable search pipeline components into `search_pipeline.{h,cpp}`:
- `GenerateTermInfos()` — N-gram generation with estimated posting list sizes
- `Execute()` — Full search pipeline (intersection, filters, verify_text)
- `SearchPipelineResult` with `empty_term_detected` sentinel
- Batch filter/PK lookups for improved throughput

### Error Propagation

- `DecodeFieldValue` returns `Expected<std::string, Error>` instead of sentinel strings
- GTID parsing uses Expected throughout
- `StartSync()` returns `Expected<std::string, Error>`

### Type Safety and Layer Decoupling

- `FilterMap` type alias with transparent hash
- `protocol_constants.h` for shared TCP protocol constants
- Handler consolidation with clear ownership boundaries

## Migration Guide

### From v1.5.4

**No breaking changes to wire protocol or on-disk formats.** Direct upgrade is safe.

**New features are opt-in:**
- BM25 scoring: Use `SORT _score` in queries (requires `verify_text: "ascii"` or `"all"`)
- Synonyms: Add `synonyms` section to config
- HIGHLIGHT: Use `HIGHLIGHT` clause in queries (requires `verify_text: "ascii"` or `"all"`)
- Fuzzy search: Use `FUZZY` clause in queries
- FACET: Use `FACET` command
- V2 dump format: Automatic — new dumps use V2, old V1 dumps still loadable

**MariaDB support:** Configure the same `mysql` section in config.yaml. MygramDB auto-detects the server type from `SELECT VERSION()`.

**Behavioral change in DUMP LOAD:** The `dump_load_in_progress` flag now stays active until after replication restart and BM25 statistics rebuild complete. Search handlers return "loading" responses for slightly longer during DUMP LOAD, but the system state is now consistent.
