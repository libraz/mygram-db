# MygramDB v1.8.0 Release Notes

**Release Date:** 2026-07-03
**Type:** Bug Fix / Correctness Hardening
**Previous Version:** v1.7.0

---

## Overview

Version 1.8.0 is a correctness- and reliability-focused release. It closes out several regressions introduced while building v1.7.0's multi-database support, hardens binlog replication against unrecoverable errors and primary-key changes, fixes a reverse-search ordering bug, and tightens HTTP/TCP query-parsing parity. It also adds date-only string support to datetime parsing and a round of client/CLI/config robustness fixes.

No breaking changes. One default-value change worth noting: the global `ngram_size` default (when omitted from configuration) is now `2` (bigram) instead of `1`.

**Highlights:**

- **Critical replication fix** — a filter-only `UPDATE` (e.g. changing a `status` column without touching indexed text) could silently drop a still-qualifying document from full-text search results
- **Critical replication fix** — `UPDATE` events that change a row's primary key now correctly reindex as delete-then-insert instead of leaving a stale document under the old key
- **Reverse search ordering fix** — `Index::SearchAnd(reverse=true)` now reliably returns highest-DocID-first, including for partial (non-truncated) result sets
- **Table-key resolution fix** — an exact registered table key now always takes priority over qualified-name fallback scanning, preventing requests from being silently routed to a different same-named table
- **Binlog fail-fast hardening** — CRC32 checksum mismatches, `TABLE_MAP_EVENT` parse failures, tagged GTIDs, and unsupported binlog configurations now trigger a reconnect or hard failure instead of continuing on inconsistent state
- **HTTP query construction rebuilt** — `q` is now always treated as literal search text (no more reserved-keyword smuggling via `q`), with case-insensitive filter/sort/facet column resolution
- **Boolean query fixes** — correct substring fallback for single-character terms, quoted-phrase handling for literal `AND`/`OR`/`NOT`, `<>` operator support, and column-case preservation
- **Reactor buffer-overflow misclassification fix** — pipelined requests no longer get incorrectly disconnected as "request too large"
- **Date-only string support** — `DateTimeToEpoch`/`ParseDatetimeValue` now accept `YYYY-MM-DD` (midnight) and truncate fractional epoch seconds
- **27 commits** since v1.7.0

## Critical Fixes

### 1. Filter-only UPDATE could remove a document from the full-text index

**Type:** Bug Fix
**Severity:** Critical

`BinlogEventProcessor::ProcessEvent`'s UPDATE handler decided whether to mutate the full-text index based on whether the after-image text was *empty*, rather than whether the text actually *changed*. A filter-only `UPDATE` (e.g. `UPDATE articles SET status = 2 WHERE ...`, not touching the indexed `content` column) whose after-image text came back empty could therefore remove a still-qualifying document from the index — the row remained in the document store, but silently disappeared from search results.

Fixed by no longer treating an empty after-image text as a signal to remove the document when it still satisfies all required filters; the existing index entry and stored text are preserved instead.

**Files Changed:**

- `src/mysql/binlog_event_processor.cpp`
- `tests/mysql/binlog_event_processor_test.cpp`

### 2. UPDATE that changes the primary key left a stale document under the old key

**Type:** Bug Fix
**Severity:** Critical

UPDATE events that change a row's primary key only ever wrote to the new key, leaving a stale, unreachable document indexed under the old key.

Fixed by detecting a primary-key change on UPDATE and splitting it into a DELETE for the old key followed by an INSERT for the new key, preserving filters on both synthesized events.

**Files Changed:**

- `src/mysql/binlog_event_parser.cpp`, `src/mysql/binlog_event_processor.cpp`

### 3. Reverse search ordering was wrong for partial (non-truncated) result sets

**Type:** Bug Fix
**Severity:** High

`Index::SearchAnd(reverse=true)` only reversed the result order after truncating to `limit`, so result sets smaller than `limit` were returned in ascending DocID order instead of descending.

Fixed to apply the reversal consistently regardless of whether truncation occurred, so `reverse=true` always returns highest DocIDs first.

**Files Changed:**

- `src/index/index.cpp`
- `tests/index/index_gettopn_test.cpp`

### 4. Table-key resolution could route a request to the wrong table

**Type:** Bug Fix
**Severity:** Critical

`ResolveTableKey` ran a database-qualified scan before checking for an exact match on the literal registered key. A client explicitly registered under a bare name (e.g. `articles`) could be silently routed to a different, same-named qualified table (e.g. `mydb.articles`) if both existed in the catalog — a data-correctness bug with real user impact.

Fixed by restoring exact-literal-match-first priority; the qualified `<database>.<table>` scan is now only used as a fallback when no exact key exists.

**Files Changed:**

- `src/server/table_catalog.cpp`

## Replication & Binlog Hardening

### 5. Fail-fast on unrecoverable binlog processing errors

Previously, a CRC32 checksum mismatch, a `TABLE_MAP_EVENT` parse failure, or a failure to fetch column names for a monitored table logged a warning and continued — risking silently missing or misattributed row events. These now trigger a reconnect from the last processed GTID. Receiving a `GTID_TAGGED_LOG_EVENT` is now treated as fatal (tagged GTIDs cannot be safely tracked for reconnect). Truncated UPDATE before/after row images are now always propagated as an error instead of being treated as a lenient end-of-rows signal.

**Files Changed:** `src/mysql/binlog_reader*.cpp`, `src/mysql/rows_parser.cpp`

### 6. Database-qualified required-table validation

Required-table existence checks previously relied on the connection's default `DATABASE()`. They now query `INFORMATION_SCHEMA` with an explicit `TABLE_SCHEMA` filter per table, so multi-database configurations validate the correct schema for each table. Unsupported binlog configurations (non-CRC32 checksum, `PARTIAL_JSON` row-value mode) are now rejected outright rather than warned about. Reconnection attempts are now serialized to reject overlapping reconnects.

**Files Changed:** `src/mysql/connection_validator*.cpp`, `src/mysql/mysql_reconnection_handler.cpp`, `src/app/server_orchestrator.cpp`

### 7. TIMESTAMP/DATE filter values parsed as raw integers

Binlog and initial-load filter extraction decoded `TIMESTAMP`/`DATE` values as raw integers instead of through the datetime parser, producing wrong values for fractional-second epoch values and date strings.

**Files Changed:** `src/mysql/rows_parser_filter.cpp`, `src/storage/initial_loader.cpp`

## HTTP / Query Correctness

### 8. HTTP query construction rebuilt

The HTTP JSON API previously concatenated `q`/`limit`/`offset` into a query-parser string and re-parsed it, allowing reserved clauses (`LIMIT`/`OFFSET`/`SORT`) to be smuggled into `q`. `q` is now always treated as literal search text via direct `query::Query` construction. Filter, sort, and facet column names are now resolved case-insensitively while preserving the caller's original casing. `SORT _score` is now rejected explicitly (instead of silently degrading) when normalized text storage is disabled.

**Files Changed:** `src/server/http_server.cpp`, `src/server/search_pipeline.cpp`

### 9. HTTP reads rejected during table synchronization

HTTP search/count/facet/get requests now return `503` while a table is synchronizing, and `/health/ready` surfaces `sync_in_progress`, instead of racing against an in-flight `SYNC`.

**Files Changed:** `src/server/http_server.cpp`

### 10. Boolean query AST fixes

- A search term producing no n-grams (e.g. a 1-character term under bigram indexing) now falls back to a normalized substring scan instead of returning an empty result set.
- A quoted token containing an embedded `AND`/`OR`/`NOT` keyword is now correctly treated as a literal phrase rather than being split into boolean operators.
- Filters gained the SQL-style `<>` not-equal operator; filter/sort column-name casing is preserved instead of being lowercased.
- `FUZZY` now rejects invalid distance values with an explicit error instead of silently ignoring the token.
- `NOT`-term n-grams are now correctly registered for cache invalidation again (an empty-posting-list cache guard added while rebuilding query construction had unintentionally also suppressed caching and invalidation registration for legitimate NOT-term exclusions).
- `SearchRaw` boolean expressions (e.g. `alpha AND (xqz OR jkv)`) are now sent unquoted from the client and correctly detected as boolean by the parser, fixing a case where nested-under-AND `OR` groups failed to parse.

**Files Changed:** `src/query/query_ast*.cpp`, `src/query/query_parser*.cpp`, `src/server/search_pipeline.cpp`, `src/client/mygramclient.cpp`

### 11. Query-length error message dropped the configured limit

The HTTP query-length-exceeded error message was generic (`"Query text exceeds maximum length"`), unlike the TCP/parser path which includes the configured limit. Restored the specific limit value for consistency and easier client debugging.

**Files Changed:** `src/server/http_server.cpp`

## Server & Concurrency

### 12. Pipelined requests misclassified as buffer overflow

The reactor connection's 1 MiB read-buffer cap was applied to the whole accumulation buffer, including bytes already extracted into complete frames. A client pipelining several small requests back-to-back could be disconnected with "request too large" even though no individual frame exceeded the limit. Frames are now extracted incrementally and the cap applies only to the unframed tail.

**Files Changed:** `src/server/reactor/reactor_connection.cpp`, `src/server/connection_acceptor.cpp`

### 13. SYNC/OPTIMIZE/snapshot concurrency guards

`SYNC` is now rejected while an `OPTIMIZE` or `DUMP SAVE`/`LOAD` is in progress, instead of racing with an index rebuild or snapshot in flight. Scheduled automatic snapshots now skip while an `OPTIMIZE` is running, and orphaned `auto_*.dmp.tmp` files left by a crashed or interrupted snapshot write are cleaned up after one hour.

**Files Changed:** `src/server/handlers/sync_handler.cpp`, `src/server/snapshot_scheduler.cpp`

### 14. Daemon-mode relative paths broken by chdir("/")

Daemonization changes the process working directory to `/`, breaking relative `dump.dir`/`logging.file` paths configured before the fork. These are now absolutized before daemonizing.

**Files Changed:** `src/app/application.cpp`, `src/config/configuration_manager.cpp`

## Storage & Config

### 15. V2 dump GTID length incorrectly bounded by the V1 path length limit

Added a dedicated 64 KB bound for V2 dump GTID strings instead of reusing the V1 path-length limit, validated before any file is written.

**Files Changed:** `src/storage/dump_format_v2.cpp`

### 16. Config validation tightened

- `ENUM`/`SET` filter types are now explicitly rejected (their labels aren't derivable from binlog row events without extra metadata, which could let initial load and replication disagree).
- **Default `ngram_size` changed from `1` to `2` (bigram)** when omitted from configuration.
- The `start_from: snapshot` requirement for `auto_initial_snapshot` now applies to single-table deployments as well, not just multi-table ones.

**Files Changed:** `src/config/config.cpp`

---

## Client / CLI

### 17. CLI DOC output escape decoding and exit-code fixes

`DOC` command output previously copied unrecognized escaped characters (`\n`, `\r`, `\t`, `\\`, `\"`, `\xHH`) through verbatim instead of decoding them. Exit-code detection for single-command responses no longer misfires on a successful payload that happens to contain disconnect-related wording.

**Files Changed:** `src/cli/*.cpp`

### 18. Client socket teardown and C API hardening

Socket state is now consistently torn down on every send/receive failure and connection-closed path. `FACET` response parsing no longer drops rows whose value legitimately starts with `#`. The C API now clears the last-error state on successful typed calls and null-checks string allocations.

**Files Changed:** `src/client/mygramclient.cpp`, `src/client/mygramclient.h`

### 19. Date-only string and fractional epoch second support

`DateTimeToEpoch`/`ParseDatetimeValue` now accept `YYYY-MM-DD` date-only strings (interpreted as midnight) and truncate `TIMESTAMP2`-style fractional epoch values to whole seconds.

**Files Changed:** `src/utils/datetime_converter.cpp`

---

## Internal Hardening

- `Expected<T, E>` and `Expected<void, E>` are now `[[nodiscard]]`, so results can no longer be silently discarded; the `MYGRAM_ERROR` macro was removed in favor of `MakeErrorWithLocation()`.
- `CacheManager`'s destructor no longer rewrites `QueryCache` eviction callbacks during teardown (the setters aren't synchronized against `QueryCache`'s own out-of-mutex callback invocation, making the rewrite itself unsafe).
- `StructuredLog::Field` guards against a null C-string pointer.

**Files Changed:** `src/utils/expected.h`, `src/utils/error.h`, `src/utils/structured_log.h`, `src/cache/cache_manager.cpp`

## Migration Guide

No migration steps required. If your configuration omits `ngram_size` (relying on the default), note that the default changed from `1` to `2` (bigram) — set `ngram_size` explicitly if you need the previous behavior.
