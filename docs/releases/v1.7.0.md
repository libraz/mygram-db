# MygramDB v1.7.0 Release Notes

**Release Date:** 2026-06-15
**Type:** Feature / Breaking Change / Bug Fix
**Previous Version:** v1.6.1

---

## Overview

Version 1.7.0 introduces **multi-database support** through a database-qualified table identity, alongside boolean search, HTTP/TCP feature parity, multi-table consistent snapshots, and GTID-coordinated dumps. A single instance can now index tables from more than one database, with each table addressed as `database.table`.

The qualified identity is **backward compatible for the common single-database deployment**: bare table names such as `articles` continue to work and resolve to the only configured database. Qualification becomes required only when the configuration spans two or more databases. HTTP table routes are consolidated under a single `/tables/{identity}/...` shape.

This release also fixes a **critical replication regression** introduced while building the qualified identity, hardens the binlog/storage parsing paths, and migrates the English/Japanese documentation to the dedicated docs site at <https://mygramdb.libraz.net>.

**Highlights:**

- **Database-qualified table identity** — `(database, name)` identity (`app_db.articles`); one instance can index same-named tables from different databases
- **Single-database backward compatibility** — Bare names keep working; qualification required only with 2+ databases
- **Unified HTTP routes** — `POST /tables/{identity}/search|count|facet` and `GET /tables/{identity}/{primary_key}`
- **Boolean search** — `AND` / `OR` / `NOT` / parentheses evaluated through a parsed AST, with structured errors for invalid expressions
- **HTTP/TCP parity** — BM25 `_score` sort over HTTP, strict `/count` field validation, FACET over HTTP
- **Multi-table consistent snapshot** — Single `TRANSACTION WITH CONSISTENT SNAPSHOT` across all tables with a shared GTID
- **GTID-coordinated dump** — Concurrent dump/snapshot operations agree on the exact drained GTID
- **Index format V3/V4** — Header carries `kanji_ngram_size` / `cross_boundary_ngrams`; atomic two-phase dump load
- **Version-gated cache inserts** — Stale inserts racing with invalidation are rejected
- **Critical binlog replication fix** — Qualified table keys propagated through the parser and reader threads
- **Documentation** moved to <https://mygramdb.libraz.net>
- **9 commits**, 411 files changed

## Breaking Changes

### 1. HTTP table routes restructured

Table routes are now a single identity segment:

- `POST /tables/{identity}/search`
- `POST /tables/{identity}/count`
- `POST /tables/{identity}/facet`
- `GET /tables/{identity}/{primary_key}`

where `{identity}` is `database.table` (e.g. `app_db.articles`) or, in a single-database deployment, a bare `table` (e.g. `/tables/articles/search`). The previous two-segment `/tables/{database}/{table}/...` routes and the legacy `/{table}/...` routes are **removed**. HTTP clients must move to the `/tables/...` paths.

**Files Changed:**

- `src/server/http_server.cpp`, `src/server/http_server.h`

### 2. Multi-database configurations require qualified table references

When two or more databases are configured, every TCP/CLI/C++/C API/HTTP table reference must use `database.table`. A bare (ambiguous) name is rejected with `Bare table names are not supported; use <database>.<table>`. Single-database deployments are unaffected.

**Files Changed:**

- `src/config/config.cpp`, `src/config/config.h`
- `src/server/table_catalog.cpp`, `src/server/table_catalog.h`
- `src/server/handlers/command_handler.cpp`, `src/server/handlers/command_handler.h`

## New Features

### 1. Database-Qualified Table Identity and Multi-Database Support

Tables carry a `(database, name)` identity written as `app_db.articles`. The effective database is `tables[*].database` when set, otherwise `mysql.database`. A single instance can index same-named tables from different databases (e.g. `live_db.articles` and `archive_db.articles`); duplicate `(database, name)` pairs are rejected at parse time, and dump metadata persists per-table databases so identities round-trip across V1 and V2 dump formats.

Bare table names remain valid in single-database deployments through a shared resolver (`ResolveTableKey`) used by both the TCP catalog and the HTTP server; the multi-database gate (`config::RequiresQualifiedTableReferences`) is centralized in the config layer.

**Files:** `src/config/config.*`, `src/server/table_catalog.*`, `src/server/http_server.*`, `src/server/handlers/command_handler.*`, `src/server/handlers/sync_handler.cpp`

### 2. Boolean Search

The search pipeline evaluates parsed `QueryNode` AST expressions via `ExecuteWithBooleanAst()`. Boolean syntax (`AND` / `OR` / `NOT` / parentheses) is detected ahead of the fuzzy/synonym paths, AST terms are collected for highlight and cache-key generation, and syntactically invalid expressions return a structured error.

**Files:** `src/query/search_pipeline.*`

### 3. HTTP/TCP Parity and FACET over HTTP

- BM25 `_score` sorting on HTTP `/search`
- `/count` rejects unsupported fields (`limit`, `offset`, `sort`, `highlight`, `fuzzy`) with 400 instead of ignoring them
- Pipeline errors return 400 (was 500); boolean queries highlight correctly via `pipeline.all_search_terms`
- `POST /tables/{identity}/facet` exposes FACET aggregation over HTTP
- `replication_paused_for_dump` surfaced in `/health/ready` and `/health/detail` so dumps no longer cause spurious health degradation

**Files:** `src/server/http_server.cpp`

### 4. Multi-Table Consistent Snapshot and GTID-Coordinated Dump

- `ServerOrchestrator` opens one `TRANSACTION WITH CONSISTENT SNAPSHOT` across all tables when replication is enabled and more than one table is configured, capturing a shared GTID before loading
- `replication_pause::Counter` gains `PublishDrainedGTID()` / `WaitForDrainedGTID()` so concurrent dump/snapshot operations agree on the exact GTID at which the reader was stopped
- `IsGTIDModeEnabled()` returns `Expected<bool, Error>` so query failures propagate instead of being treated as OFF

**Files:** `src/app/server_orchestrator.cpp`, `src/mysql/*`, `src/server/dump_handler.cpp`, `src/server/snapshot_scheduler.cpp`

### 5. Index Format V3/V4 and Version-Gated Cache Inserts

- Index header carries `kanji_ngram_size` and `cross_boundary_ngrams`
- `Index::ReplaceWithLoaded()` / `DocumentStore::ReplaceWithLoaded()` enable atomic two-phase dump load (validate all table data, then apply in bulk)
- `CacheManager` adds a `data_version_` counter with `CaptureDataVersion()` / `InsertIfVersion()` so an insert that races with a concurrent invalidation is skipped

**Files:** `src/index/*`, `src/storage/dump_format_v1.*`, `src/cache/cache_manager.*`, `src/query/search_pipeline.*`

## Bug Fixes

### 1. Critical: Binlog Events Dropped for Database-Qualified Tables

**Type:** Replication correctness
**Severity:** Critical

The binlog event parser and reader threads looked up table contexts by bare table name, but table contexts are keyed by the database-qualified identity. As a result, **every binlog event for a monitored table was silently dropped as "unmonitored"** whenever a database name was present in the config, breaking live replication entirely. The parser now populates and emits a qualified `table_key` (falling back to the bare name for empty-database configs), and the reader threads apply the same qualified-first / bare-name-fallback lookup for monitoring and error reporting.

**Files Changed:**

- `src/mysql/binlog_event_parser.cpp`
- `src/mysql/binlog_reader_threads.cpp`

## Changed / Refactor

- **Cache module consolidation** — `cache_key` and `md5` moved from `src/query/` to `src/cache/`; `DumpManager` removed (absorbed by `dump_format_v1/v2` and `document_store_persistence`)

- **Build** — `POSITION_INDEPENDENT_CODE` enabled on `mygramdb_utils` and its static dependencies

## Migration Guide

1. **HTTP clients** — Change request paths from `/{table}/...` to `/tables/{table}/...` (single database) or `/tables/{database}.{table}/...` (multi-database).
2. **Single-database TCP/CLI/client-library users** — No change required; bare table names keep working.
3. **Multi-database users** — Qualify every table reference as `<database>.<table>`.
4. **Dumps** — Keep old dumps for rollback, then create a fresh dump after upgrading so restored metadata includes qualified table identities.
