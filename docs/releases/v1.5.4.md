# MygramDB v1.5.4 Release Notes

**Release Date:** 2026-04-13
**Type:** Feature / Bug Fix / Refactor
**Previous Version:** v1.5.3

---

## Overview

Version 1.5.4 adds **CRC32 checksum verification for binlog events**, enabling end-to-end data integrity checking on received replication events. The release also delivers a broad correctness, security, and thread-safety hardening sweep across the entire codebase — fixing use-after-free races, deadlocks, undefined behavior, memory tracking underflows, and partial network I/O — along with major refactoring to split large implementation files into focused modules, deduplicate helpers, and consolidate constants.

**Highlights:**

- **CRC32 binlog checksum verification** — Each binlog event is verified against its trailing CRC32 checksum; corrupted events are logged and skipped
- **Use-after-free fix in FilterIndex** — `GetEqBitmap` now returns a copy instead of a raw pointer into internal storage
- **Deadlock fix in SyncOperationManager** — `StopSync` no longer joins threads while holding `sync_mutex_`
- **Memory tracking underflow fix** — Cache entries record heap footprint at insertion time to prevent `total_memory_bytes_` going negative
- **DP-based LIKE pattern matching** — Replaces exponential backtracking with O(n*m) dynamic programming
- **8 large files split into focused modules** — All files over 1000 lines decomposed into logically grouped translation units
- **267 files changed**, 18,321 insertions, 9,235 deletions

## New Features

### 1. CRC32 Checksum Verification for Binlog Events

**Type:** Feature
**Commit:** `2843bed`

Enable end-to-end data integrity checking on received binlog events:

- Switch `@source_binlog_checksum` from `NONE` to `CRC32` so MySQL appends a 4-byte checksum to every event
- Verify each event in `BinlogReader::ReaderThreadFunc()` by computing CRC32 over all bytes except the trailing 4 and comparing to the stored value; corrupted events are logged and skipped
- Add `crc_errors_` atomic counter and `GetCRCErrors()` accessor to `BinlogReader` for observability
- Add `kBinlogChecksumSize = 4` constant to `constants.h`, replacing all hardcoded 4-byte offsets
- Add `kMySQLBinlogChecksumMismatch` (2016) error code

**Files:**

- `src/mysql/binlog_reader.{h,cpp}` — CRC32 verification in reader thread
- `src/mysql/binlog_event_parser.cpp`, `src/mysql/rows_parser.cpp` — Use `kBinlogChecksumSize` constant
- `src/utils/constants.h`, `src/utils/error.h` — New constant and error code

**Testing:** `tests/mysql/binlog_crc32_test.cpp` — 7 tests covering CRC32 computation, valid/corrupted event verification, minimal event handling, empty data, and determinism. Tests run without MySQL dependency.

## Bug Fixes

### 1. Use-After-Free in FilterIndex::GetEqBitmap

**Type:** Bug Fix
**Severity:** High — Concurrent writers could free the bitmap while callers held the pointer
**Commit:** `dfbe2cd`

`FilterIndex::GetEqBitmap` returned a raw pointer into internal storage. Concurrent `RemoveDocument` calls could free the underlying bitmap. Changed to return `RoaringBitmapPtr` (via `roaring_bitmap_copy` under shared lock).

**Testing:** `GetEqBitmapReturnsCopy` verifies the returned bitmap survives subsequent `RemoveDocument` calls; `ConcurrentReadWriteSafety` runs 5,000 concurrent add/remove/read iterations.

### 2. Deadlock in SyncOperationManager::StopSync

**Type:** Bug Fix
**Severity:** High — Background thread's `update_state` lambda also acquires `sync_mutex_`
**Commit:** `dfbe2cd`

`StopSync` joined background threads while holding `sync_mutex_`. The background thread's update callback also acquires `sync_mutex_`, causing a deadlock. Fixed by moving threads to a local map under `sync_mutex_` and joining after releasing the lock.

### 3. SYNC Data Race and Cache Invalidation

**Type:** Bug Fix
**Severity:** High — Replication threads could re-add stale documents during SYNC
**Commit:** `1024777`

Multiple issues in `SyncOperationManager`:
- Replication was not stopped before clearing the index/doc_store during SYNC, allowing concurrent re-adds
- Stale posting lists persisted when `InitialLoader` skipped existing PKs
- Search cache was not invalidated after SYNC rebuild

Fixed by stopping replication first, explicitly clearing index and doc_store, and invalidating `CacheManager` for the synced table after rebuild.

### 4. CRC32 Stripping in QUERY_EVENT

**Type:** Bug Fix
**Severity:** Medium — Garbage bytes parsed at the end of each QUERY_EVENT
**Commit:** `1024777`

`binlog_event_parser.cpp` did not strip the 4-byte CRC32 checksum from `QUERY_EVENT` effective length, even when checksums are nominally disabled via `SET @source_binlog_checksum`.

### 5. NULL Column Parsing in Rows Parser

**Type:** Bug Fix
**Severity:** Medium — Valid NULL columns triggered truncation warnings
**Commit:** `1024777`

`ParseWriteRowsEvent` and `ParseDeleteRowsEvent` applied buffer bounds checks to NULL columns. Since NULL columns consume no bytes, reaching the buffer end is valid.

### 6. Memory Tracking Underflow in Query Cache

**Type:** Bug Fix
**Severity:** Medium — `total_memory_bytes_` could go negative
**Commit:** `7b9f333`

Vector capacity can change between insertion and removal. Added `stored_memory_footprint` field to `CacheEntry` that records heap footprint at insertion time; used in `Erase()` and `RemoveEntryLocked()`.

### 7. Filter-Column Invalidation Bypass

**Type:** Bug Fix
**Severity:** Medium — Cache entries skipped invalidation when both text and filters changed
**Commit:** `7b9f333`

`InvalidationManager` only invalidated filter-bearing entries when `changed_ngrams` was empty. When both text and filter columns changed simultaneously, filter entries were not invalidated.

### 8. Partial Network I/O in CLI and Client

**Type:** Bug Fix
**Severity:** Medium
**Commits:** `fe32ea1`, `7b9f333`, `b7b4835`

- `mygram-cli.cpp`: Handle partial sends in send loop; loop `recv()` until CRLF terminator instead of relying on a single call
- `mygramclient.cpp`: Replace single `send()` with a loop for partial writes; check `setsockopt` return values

### 9. PostingList Serialization Endianness

**Type:** Bug Fix
**Severity:** Medium — Byte order incorrect in delta and roaring serialization
**Commit:** `7b9f333`

Replaced big-endian bit-shift serialization with `WriteUint32LE`/`ReadUint32LE` helpers. Also replaced `roaring_bitmap_portable_deserialize` with `roaring_bitmap_portable_deserialize_safe` passing the buffer size.

### 10. Exponential Backtracking in LIKE Pattern Matching

**Type:** Bug Fix
**Severity:** Medium — Pathological patterns could cause O(2^n) evaluation
**Commit:** `14dffd0`

Replaced recursive backtracking in `MatchLikePattern` with O(n*m) dynamic programming.

### 11. Comprehensive Security Hardening

**Type:** Bug Fix / Security
**Severity:** Medium
**Commits:** `14dffd0`, `fe32ea1`, `659f7a1`

- Password/credential masking in `SHOW VARIABLES` output
- SQL injection prevention in `InitialLoader` and `ConnectionValidator`
- Path traversal detection fix (no longer blocks filenames like `..env`)
- JSON Schema validation and path traversal checks in config
- Port range validation (1-65535) in CLI and config
- Null byte injection rejection in config paths and bind addresses

### 12. Thread Safety Fixes

**Type:** Bug Fix
**Severity:** Medium
**Commits:** `14dffd0`, `659f7a1`, `fe32ea1`, `b7b4835`

## Migration Guide

### From v1.5.3

**No breaking changes to wire protocol or on-disk formats.** Direct upgrade is safe.

**CRC32 checksum verification** is enabled automatically. MySQL will now append CRC32 checksums to binlog events. Events that fail verification are logged at warning level and skipped. Monitor `GetCRCErrors()` / the CRC error counter for unexpected corruption rates.

**RPM upgrade behavior change:** Upgrading the RPM package no longer triggers an automatic service restart. Operators must explicitly stop and start the service during upgrade. This prevents unexpected downtime during `dnf upgrade` on HA pairs.

**Error code relocation:** Sync-related error codes moved from the network range to the business-logic range:
- `kSyncTableNotFound`: 6030 → 4010
- `kSyncAlreadyInProgress`: 6031 → 4011
- `kSyncMemoryCritical`: 6032 → 4012

If you have monitoring or alerting based on numeric error codes in these ranges, update accordingly.

**API change (C++ library only):** `CacheManager::Insert()` ngrams parameter changed from `std::set<std::string>` to `std::vector<std::string>`. `StartSync()` return type changed from `std::string` to `Expected<std::string, Error>`. These affect only internal callers.
